# Инженерный устав: критичная система (гос-уровень / highload)

Документ задаёт **обязательный** уровень для продукта, к которому предъявляются требования, сопоставимые с **критичным порталом** и **высоконагруженной** средой. Детали в коде, [`.cursor/rules/critical-system-engineering.mdc`](../.cursor/rules/critical-system-engineering.mdc), **[`.cursor/rules/portable-backend-architecture.mdc`](../.cursor/rules/portable-backend-architecture.mdc)** (слои, запрет Supabase в UI, `BACKEND_MODE`, dual impl), репозитории CI, хуки ([`docs/git-hooks.md`](git-hooks.md)).

**Целевое состояние:** ошибки **сведены к минимуму** (не «невозможны» — инженерно корректно: предсказуемо, детектируемо, с автоматикой), безопасность **enforced** политикой и инструментами, производимость **измеряема и вписана** в SLO, система **в любой момент** готова к смене бэкенда/хостинга.

---

## 0. Уровень соответствия (compliance)

| Принцип | Что значит на практике |
|--------|------------------------|
| **Defence in depth** | Клиент + сеть + API + БД: каждый слой валидирует и не доверяет предыдущему. |
| **Zero trust (данные)** | Нет «доверенного» UI; разрешения — только с сервера/политик. |
| **Fail safe** | При сбое — предсказуемая деградация, не тихий порча данных. |
| **Observability by design** | Каждое важное действие **наблюдаемо** (метрика, лог, трассировка). |

---

## 1. Процесс (до кода, каждый раз)

1. **Уточнить** требования, границы, роли, объём, офлайн, **пики нагрузки** и **последствия отказа** (BIA по упрощённой шкале).
2. **Анализ рисков:** безопасность, целостность данных, перф, масштаб, отказы, **регуляторные** ограничения (PII, хранение).
3. **Архитектура:** слои, контракты (DTO/порты), точки расширения, `BACKEND_MODE` / feature flags, **сценарии деградации**.
4. **Реализация** — только после п.1–3.

---

## 2. Enforcement (автоматика, без исключений по умолчанию)

Правила **не существуют «на бумаге»**, если нет **автопроверки** в **CI** и (где уместно) **pre-push / pre-commit**.

### 2.1 Обязательные проверки в CI

| Проверка | Назначение | Блок merge при fail |
|----------|------------|----------------------|
| **Статический анализ** | `flutter analyze` / `dart analyze` (без `error`); для backend — `tsc`/`npm run build`/линтер. | **Да** |
| **Тесты** | `flutter test` (и при наличии — интеграционные/контрактные job’ы). | **Да** (см. исключения ниже) |
| **Сборка** | Release-вариант (Android/iOS) или `flutter build` в CI, чтобы ловить compile-time. | **Рекомендовано «да»** |
| **Сканирование секретов** | `gitleaks`, `trufflehog`, `git-secrets` или аналог в **каждом** push/PR. | **Да** |
| **Политика зависимостей** | SCA (уязвимости в зависимостях) по согласованной частоте. | По policy |

### 2.2 Ветвление и защита

- **Нельзя** сливать в `main` / `release` без **зелёного** обязательного pipeline (анализ + тесты + секрет-скан).
- Для **hotfix** в прод: отдельный путь, но **тот же минимум анализа**; отсутствие тестов — только с **документированным риском** и post-merge дописыванием (не норма).

### 2.3 Локальные хуки

- Pre-commit: формат, анализ изменённого, быстрый dry-run. См. [`docs/git-hooks.md`](git-hooks.md).
- Локальные хуки **дополняют** CI, **не подменяют** его: всё критичное — на сервере.

### 2.4 Исключения

Исключение из **обязательных тестов** на PR допускается **только** с:

- ссылкой на задачу/риск,  
- **дедлайном** довнесения тестов,  
- и подписью ответственного (в теле PR).

**Исключения из анализа** — в крайних случаях, с `// ignore` и комментом **зачем** (и issue).

## 3. Безопасность (enforced)

### 3.1 Секреты и ключи

- **Запрещено** хранить в коде, в коммитах, в снимках: API-ключи, пароли, приватные ключи, long-lived service tokens, connection strings с паролем.
- Сборка: `--dart-define`, CI **secrets**, файлы `*.env` / `api_keys.json` **вне VCS** (см. [`lib/config/app_secrets.dart`](../lib/config/app_secrets.dart)).
- **Pre-merge:** pipeline с секрет-сканером; при срабатывании — **блок** и **ротация** скомпрометированного (если уже в истории — scrub по процедуре security).

### 3.2 Токены (приложение)

| Тип | Правило |
|-----|--------|
| **Access / session** | Короткоживущий; **не** логировать; в памяти/secure storage по платформе. |
| **Refresh** | Только в **secure** хранилищах; **не** в SharedPreferences в открытом виде. |
| **Static API keys** | Минимизировать; prefer OAuth/JWT/short-lived; ротация и отзыв. |

- Любой **лог/краш-репорт** — **без** заголовков `Authorization`, тел токенов, полных PII. См. [`secure_log`](../lib/core/secure_log.dart) и аналоги на сервере (redaction).

### 3.3 Сервер (истина)

- Валидация **всех** write на API; **IDOR** — явные проверки членства/ролей.
- RLS/политики БД + приложение **не** полагается только на RLS, если дублирует — ок как defence in depth.

### 3.4 Угрозы (минимум в чек-листе дизайна)

Replay, injection, mass assignment, прямой перебор id, **чужие** room/conversation в WS — учитывать в API, gateway и тестах.

## 4. Устойчивость: сценарии отказа (обязательные к проработке)

Каждая **критичная** фича с сетевым вызовом должна иметь **явное** поведение в таблице ниже (в коде — типизированные ошибки, в UX — копия/действие, без «белого экрана»).

| Сценарий | Ожидаемое поведение клиента | Ожидаемое поведение бэкенда/опс |
|----------|----------------------------|---------------------------------|
| **HTTP API недоступен** | Сообщение, retry/backoff, кэш/стейл где допустимо; **не** бесконечные спиннеры. | Health/readiness, алерты, autoscaling, rate limit 503 с Retry-After. |
| **WebSocket недоступен** | Статус «связь», exponential reconnect, **восстановление подписок**; догрузка с REST при сомнении в полноте. | Sticky/Redis adapter, таймауты, backpressure, не плодить дубликаты событий. |
| **БД недоступна** | Только read-only/офлайн, если кэш есть; **не** подтверждать успех без ответа API. | Readiness=fail, circuit breaker, очереди для асинхронщины; миграции вне пика. |
| **Частичный сбой** (одна зона) | Деградация фич, не полный крэш приложения. | Multi-AZ, реплики, RTO/RPO по классу сервиса. |

Идемпотентность **критичных** операций; дедуп по **server id** / idempotency key; повторы с **jittered backoff** — обязаны быть описаны для финансовых/юр. сценариев.

## 5. Метрики и SLO (производительность под контролем)

### 5.1 Что измеряем (минимум)

| Метрика | Где / как | Для чего |
|---------|-----------|----------|
| **FPS / jank** | Flutter: DevTools, `PerformanceOverlay`, при необходимости — Firebase Performance / custom. | UI не «ломает» доверие к системе. |
| **Latency** | Client: p50/p95 **время** до first byte / full response по ключевым API; server: p95 route time. | SLO, регрессии. |
| **Error rate** | Доля 4xx/5xx, сетевых сбоев, **не** пойманных исключений (crashlytics) по версии. | Алерты, откат. |

### 5.2 SLO (рекомендуемый шаблон)

- **Доступность** API (например 99,9% / месяц) — по классу критичности.  
- **p95 latency** на ключевых путях (логин, список, отправка) — **фиксированы в Jira/Notion** и не ухудшаются без ADR.  
- **Crash-free users** / сессий — не ниже порога по соглашению с бизнесом.

Регрессия перфа: см. [§8 Definition of Done](#8-definition-of-done-закрытие-задачи).

### 5.3 Код-ориентиры (Flutter + backend)

- Списки: не тянуть `Future` в `itemBuilder`, узкий `Selector`/`Listenable`, `const`.  
- API: пагинация, индексы, кеш с явной политикой инвалидации.

## 6. Definition of Done (задача **не** закрыта, пока)

Задача считается **завершённой** при **одновременном** выполнении:

1. **Тесты** — unit/widget/integration **по согласованному минимуму** на фичу; отсутствие тестов — только по §2.4.  
2. **Анализ** — **ноль** `error` в `flutter analyze` / в backend — эквивалент. Warnings — по policy (желательно 0).  
3. **Нет деградации перф** — сравнение с baseline: FPS/latency/память **в пределах** допуска; при сомнении — замер/скрин из DevTools, запись в PR.  
4. **Секьюрнити** — нет новых секретов в диффе; скан в зелёной зоне.  
5. **Поведение при отказе** — для новых сетевых путей: хотя бы **одна** ветка ошибки/офлайн покрыта тестом или мануальным сценарием **в PR**.  
6. **Миграция** — не добавлена **прямая** зависимость UI/Domain от Supabase/конкретного SDK без порта/фабрики (см. §9).

**Merge в защищённую ветку** — только при зелёном CI (§2).

## 7. Архитектура (обязательно, без ослабления)

| Слой | Ответственность |
|------|-----------------|
| **Domain** | сущности, бизнес-инварианты, **без** IO |
| **Repository** | сценарии, маппинг, политика повторов/ошибок |
| **DataSource** | Supabase / REST / WebSocket — **изолировано** |

- **UI** не вызывает напрямую `Supabase.*`, `Dio` для домена — только через `Repository` / порты.  
- `BACKEND_MODE=supabase|rest` — единая модель ([`kBackendModeEnv`](../lib/config/app_secrets.dart), [`BackendMode`](../lib/core/config/backend_mode.dart)).  
- Контракты стабильны; смена инфраструктуры **не** трогает Domain при неизменном контракте.

---

## 8. Миграция и независимость от Supabase (readiness)

**Требование:** продукт **должен уметь** работать **без Supabase** на выбранных контурах (VPS, другой IdP) при тех же user flows, что заложено в `BACKEND_MODE=rest` и аналогах.

- **Auth** — [`AuthPort`](../lib/core/auth/auth_port.dart); реализация сменяема (Supabase / JWT+refresh / др.).  
- **Данные** — фабрики/репозитории (например `ChatMessagesFactory`, репозитории списка чатов) — **без** `Supabase` в `build()` экрана.  
- **Контрактные** тесты: один и тот же DTO с обоих бэкендов, где применимо.  
- **Feature flags** и env — переключение без **hardcode** URL в UI.

Отсутствие «ржавого» импорта `supabase_flutter` в `lib/features/*/domain` — **ожидаемая норма**; в UI — только через сервисы/репо.

## 9. Backend (ориентир)

REST + WebSocket; PostgreSQL; Redis (при scale-out WS); JWT / доверенный IdP; OpenAPI/контракты, согласованные с Flutter. Health/ready, структурированные логи, **корреляция** (request id).

---

## 10. DevOps

`docker-compose` (локально/стадии); `.env` вне репо; CI: analyze, test, build, **secret scan**; healthchecks; согласованные **алерты** по error rate и latency. Workflows: [`.github/workflows/`](../.github/workflows/).

---

## 11. Тесты (на фичу)

| Уровень | Минимум |
|---------|---------|
| **Unit** | бизнес-логика, парсинг, валидация, идемпотентность map |
| **Integration** | repository + mock datasource, контракты |
| **Widget** | критичные формы, ошибки, деградация |

Сценарии: happy path, границы, невалидные данные, сеть/5xx, таймауты, **кратковременный** обрыв WS.

---

## 12. Self-review (после кода, обязательно)

1. Уязвимости (секреты, IDOR, ввод) — **исправить**.  
2. Узкие места (N+1, лишние rebuild, O(n) в hot path) — **оптимизировать** или в backlog **с метрикой**.  
3. Потенциальные падения (null, race, dispose) — **устранить**.  
4. Поведение при **DB/API/WS down** — **явно** или тест, или ссылка в PR.  
5. Читаемость, имена, тестопригодность.

---

## 13. Вывод в задаче (для агента / PR)

- Краткое **архитектурное** решение, **риски** и **SLO-влияние**.  
- **Код** (минимальный дифф по сути).  
- **Тесты** (или **явное** исключение по §2.4).  
- **Метрики/перф:** что замеряли, baseline vs after.  
- **Инструкции** переключения env (Supabase ↔ REST), если касается.

---

*Версия: критичный уровень (гос/портал / highload). Правка устава — через ревью архитекторa + согласование с владельцем продукта.*
